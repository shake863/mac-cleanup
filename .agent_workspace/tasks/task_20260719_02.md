## V1 设计草案(claude,2026-07-19)

### 需求还原
1. 参考 lemon-cleaner,新增「磁盘占比分析」能力:看清哪些目录占了多少空间。
2. 重点识别开发遗留:GitHub 项目编译产物、node_modules、Python venv/conda env 等——往往一次使用后永久占用。
3. 对每个占用给出建议:可删除 / 可优化(如 docker prune、git gc)/ 有价值勿动。
4. **交互铁律:不做全盘递归分析。一次只分析一个目录的一层,先定性(有无价值/是否值得下探),有价值再下探。**

### 架构定位(维持三层分离)
- **引擎(确定性)**:新增只读子命令 `clean-zd analyze <dir>`——展开指定目录**一层**,输出每个子项的体积、占父目录比例、类型定性、新鲜度信号。不递归、不删除、不下判断结论。
- **知识层**:新增「开发产物签名表」(识别 node_modules/venv/target 等的判定规则),放 rules.json 新 section 或独立内置表。
- **智能层(mac-clean skill,market-zd 仓库)**:驱动「analyze → 逐项定性 → 值得下探的再 analyze 子目录 → 触底后给建议 → 用户确认 → manifest/ignore 记录」循环。删除仍走既有 manifest+clean 通道,analyze 本身零删除能力。

### 引擎 analyze 命令设计
```
clean-zd analyze [DIR] [--json] [--top N] [--min-size MB]
```
- DIR 默认 `~`;必须在 $HOME 内(复用 safety 边界);只读。
- 输出每个直接子项:
  - `path`、`size`(递归体积,复用 paths.path_size,容忍 permission-denied)
  - `percent`:占本次分析目录总量比例
  - `kind`:`dev-artifact` | `dev-project` | `app-data` | `user-data` | `cache` | `system` | `unknown`
  - `signature`:命中的签名名(如 node_modules / python-venv / cargo-target),未命中为空
  - `rebuildable`:bool(签名表给出,如 node_modules=true)
  - `age_days`:子项自身 mtime 距今天数(廉价新鲜度信号;局限性见开放问题③)
  - `hint`:一句话线索(供 AI 参考,非结论),如「npm 依赖目录,可由 npm install 重建」
- `--top N` / `--min-size` 控制输出规模,避免 AI 上下文爆炸;默认 top 20 或 ≥10MB。
- 排序按 size 降序;JSON 与中文表格两种渲染,风格对齐 scan。

### 开发产物签名表(初版)
| 签名 | 判定 | rebuildable | 建议方向 |
|---|---|---|---|
| node_modules | 目录名=node_modules 且兄弟有 package.json | 是 | 项目不活跃可删,npm install 重建 |
| python-venv | 含 pyvenv.cfg;或名为 .venv/venv/env 且含 bin/python | 是 | 可删,重建即可 |
| conda-env | 位于 \*conda\*/envs/ 下 | 是 | conda env remove |
| cargo-target | 目录名=target 且兄弟有 Cargo.toml | 是 | cargo clean |
| js-build | 名为 dist/build/.next/out 且兄弟有 package.json | 是 | 可删,构建重生成 |
| gradle-build | 名为 build 且兄弟有 build.gradle\* | 是 | gradle clean |
| git-repo | 含 .git 的目录 → kind=dev-project | — | 不删项目,只提示其内部产物;.git 过大提示 git gc |
| xcode-derived | 已有 rules.json 覆盖 | — | 走既有规则,analyze 不重复报 |

命中签名 → kind=dev-artifact;含 .git/package.json/Cargo.toml → dev-project;`~/Library/*`按已有知识映射 app-data/cache;`~/Documents`、`~/Pictures` 等 → user-data。其余 unknown,交 AI 定性。

### 安全与风险基线
- analyze 严格只读;所有删除仍必须经 manifest add(内含 validate_target)→ clean。
- 开发产物一律 **risk=caution**:虽可重建,但重建有时间/网络成本,且位于用户项目目录内,必须逐项经用户确认,AI 不得批量登记。
- 签名判定必须带上下文锚(兄弟 marker 文件),防止误伤同名的用户目录(如用户自己建的 build 目录存了照片——有 package.json 兄弟才算)。

### 开放问题(请 codex 重点评审)
1. **体积缓存**:lemon 有 QMScanFileSizeCacheManager;我们首版是否不做缓存,靠 --top/--min-size 控制成本?大目录(如 ~/Library)单次 du 可能数十秒。折中案:进程内缓存 + 可选 `--fast`(跳过 >X 深度精确统计,标记 size 为估算)?
2. **签名表放哪**:rules.json 新 section(与清理规则同文件但语义不同)vs 独立 signatures 内置常量(如 dev.py 的 KNOWN 模式)?我倾向独立内置表,首版不做用户可配。
3. **age_days 语义**:目录自身 mtime 不反映深层变化。对 dev-artifact,更有意义的是**兄弟项目的活跃度**(如项目根下 .git/HEAD 或 marker 文件 mtime)。是否 V1 就按「签名命中时改用项目锚点 mtime」实现?
4. **与既有 scan dev 类别的关系**:dev.py 的 KNOWN 全局路径(pip/cargo/m2)与 analyze 的项目内产物是两个维度,保留两者还是让 analyze 吸收 dev.py?我倾向保留 scan(主动候选推送)+ analyze(交互式追问)双入口,签名知识共享。
